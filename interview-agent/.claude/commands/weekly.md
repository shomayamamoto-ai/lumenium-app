# /weekly — 週次 KPI レポート

過去 7 日間の活動を集計し、**KPI ・分類内訳 ・未消化タスク ・改善仮説** を 1 枚レポートにまとめます。毎週月曜朝に実行するのが理想。

## 引数
- `$1` : 基準日（省略可、未指定時は今日）。この日から 7 日前までを集計範囲にする。

## 手順

### Step 1: 期間決定
Bash で `date` を使い、基準日と 7 日前の日付を計算。

### Step 2: 全候補者をスキャン
- `candidates/*/profile.md` を Glob で列挙
- 各候補者について以下を抽出:
  - 面談予定日時（profile.md）
  - brief.md の有無＆生成日時
  - minutes.md の有無＆生成日時
  - classification.md から分類・確度
  - followup-messages.md の有無
  - classification.md のフォロー TODO 状態

### Step 3: KPI 集計
以下を計算:

| 指標 | 計算方法 |
|------|---------|
| 新規候補者数 | profile.md 作成日時が期間内の候補者数 |
| 面談実施数 | minutes.md 作成日時が期間内の候補者数 |
| 分類完了数 | classification.md 作成日時が期間内の候補者数 |
| 本気層率 | 本気層 / 分類完了数 |
| 未着手 TODO 数 | 期日 <= 今日 かつ 状態「未着手」の総数 |
| 期日切れ TODO 数 | 期日 < 今日 - 3 日 の TODO |

### Step 4: ボトルネック検出
- 面談予定はあるが brief.md が無い候補者数 → 「ブリーフ生成漏れ」
- minutes.md はあるが classification.md が無い → 「分類漏れ」
- classification.md はあるが followup-messages.md が無い → 「フォロー漏れ」
- 期日切れ TODO 数 > 3 → 「フォロー運用破綻リスク」

### Step 5: トレンド分析（議事録テキスト横断）
全 minutes.md を grep で横断し、以下を自動抽出:
- **頻出する「刺さった訴求」**（✅ 刺さった訴求 セクションから上位 3 個）
- **頻出する「刺さらなかった訴求」**（❌ セクションから上位 3 個）
- **頻出する「障壁」**（障壁テーブルから上位 3 個）
- **頻出する `[未確認]` 項目**（ヒアリング設計の改善候補）

### Step 6: 改善仮説提案
Claude がこれらのデータから **来週やるべきこと 3 つ** を具体的な文言で提案:
- 例: 「『価格が高い』障壁が 4/7 件。分割プラン提示の標準化を検討」
- 例: 「『保護者同意』が未確認の候補者が多い。ヒアリング項目を前倒しで聞く」

### Step 7: レポート出力
`reports/weekly-YYYY-MM-DD.md` に Write（`reports/` ディレクトリがなければ作成）。ユーザーには要約 10 行で報告:

```
📊 週次レポート（{{開始日}} 〜 {{基準日}}）: reports/weekly-{{基準日}}.md

【KPI】新規 N / 面談 M / 分類 L / 本気層率 X%
【未処理】期日切れ TODO n 件、フォロー漏れ m 件
【最も刺さった訴求 TOP1】「○○」
【最大障壁 TOP1】「○○」
【今週の提案 1】...
【今週の提案 2】...
【今週の提案 3】...

次のアクション:
  /today でフォロー漏れ解消
  /compare <成約者> <未成約> で勝ちパターンを言語化
```

## 制約
- 候補者数が 0 件の場合は「分析対象なし」と返す
- 集計は `_example/` を除外
- 個別候補者の氏名をレポートに列挙しない（数と傾向のみ、プライバシー配慮）
- ファイル作成日時は `stat -c %y` か git log で取得

# /compare — 候補者差分分析（勝ちパターン言語化）

成約者と非成約者、または複数候補者を並べて **何が差を生んだか** を言語化します。PDCA の核心コマンド。

## 引数
- `$1` 以降: 候補者 slug を 2 つ以上（例: 成約 1 名 + 未成約 2 名）

## 手順

### Step 1: 各候補者の情報を集める
各 slug について以下を Read:
- `brief.md` — 面談前仮説
- `minutes.md` — 面談実態
- `classification.md` — 分類結果
- （あれば）成約状態を示すラベル（ユーザーから明示的に指示された場合、または `candidates/<slug>/outcome.md` から）

### Step 2: 比較テーブル生成
以下の軸で横並びに:

| 軸 | 候補者 A | 候補者 B | 候補者 C |
|----|---------|---------|---------|
| 動機の具体度 | 高/中/低 | ... | ... |
| 目標期限の明確さ | あり/なし | ... | ... |
| 業界理解 | 深/中/浅 | ... | ... |
| 決裁者 | 本人/保護者/... | ... | ... |
| 予算許容 | 上限金額 | ... | ... |
| 最大障壁 | テキスト | ... | ... |
| 刺さった訴求 | テキスト | ... | ... |
| 面談中の主導性 | 高/中/低 | ... | ... |
| 次アクションの主体 | 本人/こちら | ... | ... |
| 分類結果 | ... | ... | ... |
| 結果 | 成約/未成約/保留 | ... | ... |

### Step 3: 差分からの仮説抽出
以下のパターンをレポートする:

**成約群に共通する特徴（勝ちパターン候補）**
- 箇条書きで 3–5 個。根拠となる逐語引用を議事録から 1 つずつ付ける

**未成約群に共通する特徴**
- 箇条書きで 3–5 個

**ブリーフ（面談前仮説）と実態の差**
- 面談前に予測した障壁が実際に出たか、外れたか
- ブリーフの精度向上のための観点

**面談運用の改善提案**
- 例: 「成約群は全員、面談冒頭 5 分で『期限のある具体目標』を語っている。
  未成約群は質問されるまで語らない。冒頭でこの質問を前倒ししては?」

### Step 4: 保存と返却
- `reports/compare-YYYY-MM-DD.md` に Write
- ユーザーには以下 6 行で報告:
  - 比較対象: N 名
  - 勝ちパターン仮説 TOP 1: ○○
  - 敗因仮説 TOP 1: ○○
  - ブリーフ精度の気づき: ○○
  - 次週の運用改善提案 1: ○○
  - レポート: reports/compare-YYYY-MM-DD.md

## 制約
- 候補者が 2 名未満なら「比較には 2 名以上必要」と案内して終了
- 成約/未成約のラベルがない場合は **分類結果（本気層 vs それ以外）で代用**する旨を明記
- 差分抽出は **議事録の実発言ベース**（創作禁止）
- 個人が特定されるセンシティブ情報（家族構成の詳細、健康状態など）は抽象化してレポートする
